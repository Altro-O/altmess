# Архитектура Altmess

## Обзор системы

Altmess - это веб-мессенджер, состоящий из двух основных компонентов:
1. Веб-клиент (Next.js)
2. Серверная часть (Node.js + Java)

## Архитектурные слои

### 1. Клиентский слой (Frontend)

**Технологии:** Next.js 14, React 18, TypeScript

#### Компоненты:
- `AuthProvider` - управление состоянием аутентификации
- `Navigation` - навигационная панель
- `VideoCall` - компонент видеозвонков
- `ChatPage` - основной интерфейс чата

#### Службы:
- `CryptoService` - шифрование и дешифрование сообщений
- `WebSocketService` - работа с WebSocket соединением
- `APIService` - взаимодействие с REST API

### 2. Серверный слой (Backend)

#### Node.js WebSocket-сервер
**Технологии:** Express, ws

Функции:
- Управление WebSocket соединениями
- Аутентификация клиентов
- Маршрутизация сообщений между пользователями
- Обработка событий чата

#### Java HTTP-сервер (JAVE)
**Технологии:** Java 23, Maven

Функции (в разработке):
- Обработка REST API запросов
- Управление пользователями
- Хранение истории сообщений
- Аутентификация

### 3. Уровень безопасности

#### Шифрование
- Клиент-сервер: TLS/WSS
- Сообщения: AES-256 шифрование (сквозное в перспективе)
- Пароли: SHA-256 хеширование

#### Аутентификация
- JWT-токены
- Хранение токенов в localStorage
- Проверка сессии на каждом запросе

## Диаграмма взаимодействия

```
[Browser] ↔ [Next.js Client] ↔ [Node.js WebSocket Server] ↔ [Java HTTP API Server]
                            ↕
                      [Crypto Service]
                          ↕
                      [LocalStorage]
```

## Потоки данных

### Регистрация пользователя:
1. Клиент → REST API: отправка данных регистрации
2. Java-сервер: проверка и сохранение пользователя
3. Клиент: получение JWT токена и ключа шифрования

### Аутентификация:
1. Клиент → REST API: отправка кредов
2. Java-сервер: проверка и выдача JWT токена
3. Клиент: сохранение токена и подключение к WebSocket

### Отправка сообщения:
1. Клиент: шифрование сообщения
2. Клиент → WebSocket: отправка зашифрованного сообщения
3. Node.js-сервер: маршрутизация к получателю
4. Получатель: получение и дешифрование сообщения

### Видеозвонок:
1. Инициатор → WebSocket: запрос на звонок
2. Получатель: получение уведомления о входящем звонке
3. WebRTC: установка P2P соединения для передачи медиа

## Безопасность

### Клиентская безопасность
- Все сообщения шифруются перед отправкой
- Токены хранятся в localStorage (в продакшене рекомендуется использовать httpOnly cookies)
- Проверка аутентификации перед каждым действием

### Серверная безопасность
- Проверка токенов на каждом запросе
- Ограничение частоты запросов (rate limiting)
- Валидация входных данных

## Масштабируемость

### Текущая архитектура
- Одиночный WebSocket-сервер
- Нет кластеризации
- Все данные хранятся в памяти

### Планы масштабирования
- Введение Redis для хранения сессий
- Кластеризация WebSocket-серверов
- Балансировка нагрузки
- Хранение сообщений в базе данных

## Технические ограничения

### Клиент
- Совместимость с современными браузерами
- Ограничения WebRTC в разных сетях

### Сервер
- Отсутствие persisted хранилища (в текущей реализации)
- Ограничения WebRTC для большого числа участников звонка
